üß™ Test Cases Explained

üìÑ Test Case 1: Example UI Tests

File: tests/example.spec.ts
What it does:

These are sample UI automation tests that validate the Playwright documentation website. They test basic web functionality like page navigation and element interaction.

Test Steps:
  1. Test: "has title"
    • Navigate to https://playwright.dev/
    • Wait for page to load
    • Verify that page title contains "Playwright"
    • ‚úì Test passes if title is correct
  2. Test: "get started link"
    • Navigate to https://playwright.dev/
    • Wait for page to load
    • Find and click the "Get started" link
    • Verify that navigation leads to "Installation" page
    • ‚úì Test passes if "Installation" heading is visible

üîå Test Case 2: API Testing

File: tests/mockApi.spec.ts
What it does:

This test validates API endpoints without a UI. It makes HTTP requests to a public API (reqres.in) and verifies the response data.

Test Steps:
  1. Test: "API GET request test"
    • Make a GET request to https://reqres.in/api/users?page=2
    • Verify HTTP response status is 200 (success)
    • Parse JSON response body
    • Check that first user's email is "michael.lawson@reqres.in"
    • ‚úì Test passes if email matches expected value

üé® Test Case 3: Sample UI Test

File: tests/sampleUI.spec.ts
What it does:

Another UI automation test that validates the Playwright documentation website with the same scenarios as example.spec.ts.

Test Steps:
  1. Navigate to https://playwright.dev/
  2. Verify page title contains "Playwright"
  3. Click on "Get started" link
  4. Verify "Installation" heading appears on the next page

üîß Troubleshooting

Issue: "npm command not found"

Solution: Install Node.js from https://nodejs.org. Then restart your terminal.

Issue: Tests fail with browser errors

Solution: Run npx playwright install-deps to install system dependencies, then npx playwright install to download browsers.

Issue: Cannot connect to API

Solution: Check your internet connection. The tests use https://reqres.in which requires internet access.

‚úÖ Best Practices

Use Page Objects

Create reusable page objects for complex UIs to keep tests maintainable

Use Selectors Wisely

Prefer role-based selectors over class names for stability

Add Explicit Waits

Use waitFor to handle async operations properly

Keep Tests Independent

Each test should be able to run in isolation

Use Meaningful Names

Test names should clearly describe what they validate

Review Test Reports

Always check HTML reports to understand failures
